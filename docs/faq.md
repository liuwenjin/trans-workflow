# FAQ

## 🎯 定位与适用场景

### Q: Trans-Workflow 适合哪些人使用？

Trans-Workflow 专为追求**高效率、高业务掌控度**的团队及个人设计，典型用户群体包括：

* **内容创作者与自媒体团队**：如公众号作者、自媒体矩阵、数字化营销团队、短视频工业化生产团队，用于构建标准的内容全自动/半自动生产流水线。
* **AI 应用开发者与创业团队**：如独立开发者、Agent 开发者、AI 初创团队，用于快速验证多 Agent 协作流程并交付 MVP（最小可行性产品）。
* **企业数字化转型团队**：用于企业内部搭建私有化 AI 工作流、企业知识库系统、智能办公助手、自动化文档生成系统以及各类业务流程自动化（BPA）场景。

## ⚖️ 竞品差异与产品优势

### Q: Trans-Workflow 与 Dify、n8n、Langflow 有什么区别？

Trans-Workflow 在设计哲学上更侧重于“人机协同”**与**“过程可干预”。以下是核心差异对比：

| 维度 | Trans-Workflow | Dify | n8n | Langflow |
| --- | --- | --- | --- | --- |
| **核心定位** | **Agent 协同与内容生产** | AI 应用开发与 RAG 平台 | 传统企业系统自动化平台 | LLM 实验与 Prompt 编排 |
| **连接对象** | Agent ↔ Agent ↔ 人 | 知识库、API、LLM 模块 | 系统 A ➔ 系统 B ➔ 系统 C | LLM 基础算子、模型组件 |
| **最大特点** | **运行中途结果可编辑、节点可回溯** | 完善的 API 服务与 RAG 引擎 | 强大的三方系统数据同步 | 偏向底层的开发测试与实验 |
| **人机协同** | 极其深入，强调“人把控质量” | 较弱，多为单向自动化流 | 仅支持传统审批式人工节点 | 较弱，主要面向开发者 |

## 🛠️ 核心功能与技术特性

### Q: 为什么 Trans-Workflow 极度强调“中间结果可编辑”？

在真实的生产环境中，AI 生成的结果往往无法一步到位。

> **传统工作流的痛点**：`选题 ➔ 大纲 ➔ 正文`。如果“大纲”阶段出现偏差，用户通常需要修改 Prompt 并**彻底重新运行**整个工作流，造成算力和时间的极大浪费。

**Trans-Workflow 的解决方案**：在工作流执行过程中，用户可以直接介入并**就地修改**大纲内容，点击确认后，后续的“正文生成”Agent 将基于你修改后的完美大纲继续运行。真正实现 **“AI 负责批量生产，人类负责黄金标准质量把控”**。

```text
 [选题 Agent] ➔ [大纲 Agent] ➔ ✋ [人类直接编辑/修正大纲] ➔ [正文 Agent]
```

### Q: 什么是“节点级回溯”？它有什么价值？

节点级回溯允许用户从工作流的**任意中间步骤**重新触发执行。
若一个工作流包含 `A ➔ B ➔ C ➔ D` 四个步骤，当发现 C 的生成结果不满意时：

* ❌ **无需**重新执行无辜的 A 和 B 节点。
* 你只需一键**重新运行 C**（或修改 C 的参数/上下文后继续向下执行 D）。

这一特性显著降低了长流程工作流的 Prompt 调试成本与 Token 消耗。

### Q: 是否支持人工介入与多 Agent 协作？

* **完美支持人工介入**：用户可以在任意节点进行“查看结果、编辑内容、手动确认、重新执行”四项黄金操作，这是 Trans-Workflow 的底层核心设计理念。
* **原生支持多 Agent 协作**：支持在一个工作流中编排职责不同的 Agent。
```text
[需求分析 Agent] ➔ [任务规划 Agent] ➔ [内容生成 Agent] ➔ [质量检查 Agent] ➔ [交付输出]
```

### Q: 工作流中的数据流转与长文本处理机制是怎样的？
* **无感上下文传递**：默认情况下，前序节点的输出会自动映射为后续节点的输入，系统自动维护全局拓扑上下文，用户无需手动复制粘贴。
* **长文本优化**：底层采用**数据引用机制**进行结果传递，避免了传统文本传递导致的吞字或上下文丢失问题，完美适配长文写作、深度研究报告、行业课件生成等长文本场景。

## 📦 架构、部署与数据安全

### Q: 部署 Trans-Workflow 需要准备数据库或 Docker 环境吗？
**完全不需要。**
Trans-Workflow 追求极简的 **“零依赖（Zero-Dependency）”** 设计目标。你无需配置、安装和维护任何关系型数据库（如 MySQL、PostgreSQL）或缓存系统（如 Redis），也无需理解 Docker 容器化技术，直接运行即可，对个人用户及轻量化企业团队极为友好。

### Q: 什么是“单文件部署”？支持私有化吗？
* **单文件部署**：Trans-Workflow 支持将您编排好的工作流、执行逻辑以及运行交互界面，**整体打包并导出为一个独立的 `.html` 文件**。
* **100% 私有化**：该 HTML 文件拥有完全的自主控制权，您可以将其直接拖入企业内网服务器、静态网站托管服务（如 GitHub Pages、Vercel）、云服务器或本地电脑中直接双击运行。
* **数据完全归属于用户**：由于采用无状态及单文件架构，所有工作流数据与执行结果均沉淀在用户本地，平台不留存任何业务数据，确保企业资产的绝对安全。

## 💰 权益与生态

### Q: 节点数量、执行次数是否有限制？如何收费？
* **零限制**：Trans-Workflow 自身**不限制**任何工作流节点数量，也**不限制**工作流的运行与执行次数。
* **成本构成**：工作流执行时消耗的 Token 成本完全取决于您接入的 AI 模型服务商。
* **收费模式**：平台目前的核心功能（包括工作流编排、Agent 串联、节点回溯、单文件导出及私有化部署）**永久免费**。

### Q: 项目是否开源？支持企业定制开发吗？
**支持。** 项目采用开源模式运行。企业团队或资深开发者可以自由查看源码、修改内部逻辑并进行二次深度开发（例如对接企业内部 ERP、CRM 等系统），构建专属于企业自身的智能化工作流资产。
